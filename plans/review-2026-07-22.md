## 1. Verdict

**The engine is genuinely well-built. Nothing turns it on.**

Everything merged is strong: 273 commits, 47 migrations (auto-applied to prod), 150 vitest files /
1217 cases green, 43 attack-suite blocks (352 assertions) running on real Postgres in CI, live
browser E2E green against production data, CI fully green, and a layered review discipline that has
repeatedly caught real bugs before merge (race mapping, unbounded bookings, PIN-wipe regression,
the 0047 waitlist guard). The security architecture — RLS-on-everything, append-only ledgers with
machine-enforced grants, definer-RPC-only money writes, the zero-Supabase member client — held up
under adversarial audit with only enumerable, fixable exceptions.

But the audit surfaced a category of problem no amount of per-unit verification catches:
**system-level wiring that no unit owns.** The most important findings are not bugs in any file —
they are absences between files:

1. **Nothing enqueues the recurring cadence.** The scheduler tick only *claims* jobs; no code path
   ever seeds `glofox.sync.all`. Every cadenced behavior — Glofox sync, reconciliation, deletion
   detection, segments, briefing, Stripe inbox/outbox drains, verify_money, dunning, sweeps —
   runs only when hand-fed.
2. **Live proof:** production has been frozen since **2026-07-18 19:16 UTC** (last sync_run and
   reconciliation; jobs queue 0 queued / 0 dead), with **3 warning alerts sitting unacknowledged**
   — and no alarm anywhere, because alerting is pull-only and there is no staleness watchdog.
   This is the 10-week-silent-freeze failure mode the watermark law was built against, happening
   now, during the parallel run that is supposed to be building trust.
3. **The docs a fresh model would follow were pointing at the wrong work** (fixed in this pass).

None of this diminishes what's built — it defines what the next waves must be.

### P1 — cutover-blocking build gaps

- **Authority flips change no runtime behavior** — nothing outside routes/authority.ts reads
  `current_authority`; after a real flip, Glofox import would keep overwriting Kelo-authoritative
  rows and reconciliation would alarm on by-design divergence. `/health` additionally reports a
  hardcoded 5-capability constant that contradicts 0042's 8-domain ledger. No freeze-window /
  final-import / inventory-partition machinery exists (grep `write_freeze` = 0).
- **The waiver mass re-sign event nobody planned**: 0 signatures on file, waivers are not a synced
  entity, so ~258+ active members must sign fresh — and the pre-arrival link mechanism is
  half-shipped (`waiver_link_tokens` + `enqueue_waiver_links` exist; no worker processor, no
  minting route, and the enqueued job kind would dead-letter). Blocked upstream by lawyer review
  of waiver text (legal 4c).
- **No subscription-migration machinery**: nothing bulk-creates Stripe customers/subscriptions
  with correct billing anchors; the plan's "cohort double-billing test proves impossibility" gate
  has no corresponding tool.
- **Member surface**: cancel-booking UI missing while the panel *promises* "cancel up to 12 hours
  before" (API + client function exist, tested — one wired control away); `needs_resolution`
  claimers are stuck in a signin↔account bounce loop (claim-status endpoint built, no UI; no
  operator resolution workspace; desk claim codes unminted); no client calls `/auth/refresh`
  (0045 rotation is client-side dead code — and the §14.3 60s grace window was never built, so
  first mobile use risks spurious family revocations); OTP/session purge processors never
  registered (unbounded auth-table growth).
- **Operator surface**: no People index/profile (a person exists only as a booking typeahead —
  full-stack gap, API included; operators would keep Glofox open, defeating cutover);
  `/reports/revenue` has zero consumers; marketing-screen is the last non-injectable screen and
  fully untested — on the surface that gates real sends; all 22 `/scheduling/*` routes have zero
  HTTP-layer tests.
- **9 authenticated-callable SECURITY DEFINER RPCs have zero attack coverage** — including
  money-minting `grant_gift_card` and PII-destroying `pseudonymize_person` (also: publish_sessions,
  request_person_export, record_step_up_attempt, step_up_status, freeze_hold, release_hold,
  decline_waitlist_offer, enqueue_waiver_links). `webhook_events` (no tenant_id → invisible to the
  generic guard) has no deny-all probe.
- **Member cutover comms don't exist** (no claim/switch/re-card templates anywhere) and no
  **Glofox final-archive** task exists (anything outside the 6 synced entities dies with the
  contract).

## Addendum (2026-07-22, same day) — import-integrity deep dive (owner-triggered)

The owner spotted four owner accounts showing 0 credits that should have had 6+. Investigating
(seven-agent audit + live probes) revealed a real, bounded data gap and, more valuably, two
systemic defects. **Verified against live data + code:**

- **Genuinely missing:** credit_ledger is short **41 members / 331 outstanding credits** (all 4
  owners included), and **1 active membership plan** never re-synced. Everything else is intact:
  bookings (1,000 quarantined → all present), members (701 → all present), **transactions (78 →
  all present; my first "78 missing" was a wrong-id-field query error, corrected)**.
- **NOT a schema bug.** Every over-strict contract (credits `membership_id`, members, bookings,
  transactions) was already relaxed to nullish/nullable. Members/bookings/transactions self-healed
  when their idempotent syncs re-ran a full pass. **Credits didn't** — its sync is O(members),
  per-user, rate-limited, chunked, self-re-enqueuing, and a full pass never completed after the fix
  (only 31 of 71 comp/drop-in credits re-mapped). Root cause = the frozen cadence (P0-A). Fix =
  run `glofox.sync.credits` to completion; the mapper reconstructs grant+debits faithfully. No
  data lost — raw zone (`glofox_raw`) holds every original payload.
- **Two systemic defects (scheduled as R0.8b):** (1) `strictRow` hardcodes `external_ref=NULL` on
  every parse-failure quarantine, so 121 credit quarantine rows couldn't be tied to the 331-credit
  balance shortfall — the loss was un-attributable. (2) No quarantine-rate alarm and no
  raw→derived conservation check, so a run that quarantined 71 records reported `success` silently.
  These two, plus the R0.2 watchdog, are the structural guards that convert this class of failure
  from invisible to a red alert.
- **The good news:** the two-zone quarantine architecture worked exactly as designed — it caught
  every variant and recorded why. The failure was operational (frozen cadence, un-worked queue,
  no tripwire), not a silent data-loss in the pipeline.
